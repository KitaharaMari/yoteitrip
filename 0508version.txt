一、项目概述
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
模块化旅游日程规划工具。去中心化、非线性"积木式"规划，支持分钟级精确时间轴。
纯客户端 Web App，无传统后端，数据存 localStorage（游客模式）或 Firebase Firestore（登录模式）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

地图 API:   Google Maps / Routes API v2 / Directions API / Places API / Static Maps

hooks/
  useTransitRoute.ts      ← 核心通勤时间 hook
    useTransitRoute(from, to, mode, mapsLoaded, departureTime?) → RouteResult | null
    优先 server-side 代理（Routes API v2 → Directions REST）
    fallback: Google Maps JS SDK（TRANSIT → DRIVING fallback）
    deptTs 作为 stable dep，规避 Date 对象每次新建的 effect 重触问题
    departureTime 在 effect 内从 deptTs 重建 (dept = deptTs > 0 ? new Date(deptTs) : undefined)
    cache: mode + placeId 键，设有 departureTime 时跳过缓存

                            模块级 Map 缓存，相同 lat+lng+date 不重复请求

五、核心逻辑说明
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【通勤时间计算 (useTransitRoute)】
1. 调用 /api/transit 服务端代理（绕过 CORS）
   POST → Google Routes API v2（带 departureTime ISO 字符串）
   GET  → Google Directions REST API（fallback，带 departure_time Unix 秒）
2. 服务端自动修正过期时间戳：若 departureTs ≤ now，+7天整数倍（保留星期特征）
3. 客户端 buildDepartureDate：isoDate 在过去时也 +7天整数倍
4. JS SDK fallback：TRANSIT 失败 → 自动 fallback DRIVING

【日期级联 (updateDay)】
trip.days.map((d, i) => {
  if (i === dayIdx) return { ...d, ...updates };
  if (updates.date) {
    const offset = i - dayIdx;  // 负数 = 往前，正数 = 往后
    const next = new Date(y, mo-1, dy + offset);
    return { ...d, date: iso };
  }
  return { ...d, date: undefined };
})
// 改任意 day 的日期，前后所有 day 同步偏移

【addDay 日期继承】
const lastDay = trip.days[trip.days.length - 1];
if (lastDay?.date) newDay.date = lastDay.date + 1天

【封面图流程】
创建行程 → 城市选择 → Places API photos[0].getUrl() → coverPhotoUrl
更换封面 → CoverPickerModal → 用户上传 → canvas resize(960×540, JPEG 82%) → base64 → coverPhotoUrl
TripCard: coverPhotoUrl ? <img> + bg-black/35 scrim : gradientFor(index)

【Firebase 同步流程】
useFirestoreSync: 监听 trips/currentTripId/wishlist → 2s debounce → saveCloudData
AuthProvider.onAuthStateChanged:
  user 登录 → loadCloudData
  云端空 → 上传本地
  本地空 → 加载云端
  双端有内容 → SyncConflictModal

七、已知限制 / 待优化
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 时区问题：buildDepartureDate 使用浏览器本地时区，非目的地时区
   影响跨时区旅行的通勤时间估算精度（如温哥华用户规划东京行程）
   待优化：抓取 Google Time Zone API，存入 Trip.baseLocation.timezone

2. 天气仅支持 ≤16天预报，超出范围不显示（Open-Meteo 限制）

3. localStorage persist key 已从 'traveline-trip' 改为 'yoteitrip-trip'
   旧数据不自动迁移（需手动在 DevTools 复制 localStorage key）

4. base64 封面图存 localStorage/Firestore，大图可能影响存储配额
   每张图约 50-80 KB（960×540 JPEG 82%）

5. Google Maps Place Photo URL（来自 NewTripModal 自动抓取的封面）
   可能在数月后过期（photo reference 有效期限制）

6. 导出 QR 仅包含精简行程数据（无照片、开业时间、路线），足以导入但缺少展示数据

7. 多设备实时同步：目前为 2s debounce 单向写入（非 Firestore onSnapshot 实时监听）
   两个设备同时编辑同一行程会覆盖先写入的一方
