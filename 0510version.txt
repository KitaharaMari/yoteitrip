【行程规划核心】
✅ 积木式日程模块：景点(STAY)、餐饮(MEAL)、住宿(ACCOMMODATION)、交通(TRANSPORT)、城际移动(LONG_DISTANCE)

✅ 行程总览 Tab：总天数 / 总里程 / 行程地点 / 预估总费用 + 逐日明细

✅ 通勤时间级联更新活动开始时间

────────────────────────────────────────────────────────────────
三、已知问题 / 待完善
────────────────────────────────────────────────────────────────

【功能层面】
- 天气模块与备用方案卡的定位：天气仍在（基于 Open-Meteo），但新手引导 slide 已替换为介绍备用方案；两者共存，未来可考虑天气模块开关
- 步行通勤模式：仅用 JS SDK 计算，无服务端缓存，频繁刷新可能消耗 Maps API 配额
- 多城市行程：目前 baseLocation 是单城市锚点，跨国/跨城行程的搜索范围感知较弱
- 总览页费用汇总：transit 货币 label 取第一个有 transitFareCurrency 的活动，混合货币行程（如跨国）显示会不准
- 离线支持：Vercel 部署为标准 SSR，无 Service Worker，断网时 API 功能失效
- 图片存储：封面图以 Base64 存 localStorage / Firestore，大图可能导致存储超限（~5MB）

【体验层面】
- 搜索 Overlay 仅支持 Google Places 搜索，无手动输入坐标或地址解析
- 活动卡片的「开放时间」状态（CLOSED/CLOSES_SOON）显示的是英文，需要多语言化
- DayStatsBar（油费估算）未接入多语言

【技术债】
- Firebase App Hosting 不支持 Next.js 16，仍用 Vercel 部署（Firebase 配置文件保留备用）
- Zustand persist 的 merge 策略：新增 Activity 字段（如 transitFareCurrency）在老数据中为 undefined，依赖运行时 effect 补写
- ExportModal 中 TripSummaryCard 用 inline style（html2canvas 不识别 Tailwind），维护稍繁琐
